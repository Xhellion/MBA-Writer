# Logic Checker — 段落逻辑检查器

## 触发条件

用户输入包含以下关键词时激活：
- "检查逻辑" / "帮我审一段" / "逻辑检查"
- 用户粘贴草稿并要求"帮忙看看"
- 在写作流程中自动调用（建议每写完一节检查一次）

## 检查六维度

### 1. 现象描述完整性

- ✅ 5W1H 是否完整（什么现象、何时、何地、涉及谁、为什么、如何）
- ✅ 数据/事实是否可验证
- ⚠️ 模糊表述（"大幅提升"、"明显改善" → 需要量化）
- ⚠️ 主观判断替代客观描述

### 2. 理论支撑

- ✅ 是否有明确的理论引用
- ✅ 理论是否与现象匹配
- ⚠️ 理论使用是否准确（比对理论库定义）
- ✗ 有现象无理论（建议引入XX理论）
- ✗ 有理论无分析（引用了理论但没有展开）

### 3. 论证链条完整性

```
现象 → 理论 → 分析 → 结论
 ↑       ↑      ↑      ↑
描述     lens   推导   ？ ← 检查是否完整
```

- ✅ 四步是否完整
- ⚠️ 是否存在"跳步"（现象→结论，缺少理论和分析）
- ⚠️ 是否存在循环论证
- ✗ 论证链条断裂

### 4. 结论与前文呼应

- ✅ 结论是否回答了本段提出的问题
- ⚠️ 结论是否超出本段范围（"章节越界"）
- ✗ 结论与论证无关

### 5. 学术语言规范

- ✅ 术语使用是否准确
- ⚠️ 口语化表达 → 推荐学术替换
- ⚠️ AI 典型用语（"delve into"、"crucial" 等）

### 6. 引用规范

- ✅ 引用格式是否正确
- ⚠️ 缺少引用（主张没有出处）
- ⚠️ 引用过多（一段 5+ 引用 → 建议精简）

## 输出格式

```
## 逻辑检查报告

### ✅ 通过项
- 现象描述完整（5W1H 覆盖）
- 结论与前文呼应

### ⚠️ 建议改进
- 缺少理论支撑：建议引入"服务利润链"理论解释员工满意→用户满意→企业利润的关系
- 需要量化：原文"用户满意度大幅提升" → 建议改为"用户满意度从2021年的78%提升至2024年的92%（提升14个百分点）"

### ✗ 问题项
- （如有则列出，包括具体位置和改写示例）
```
